# 🔁 agent-ops — 반복작업·비용·보안 운영 원칙

## 1. 반복작업 종료조건 먼저 선언
막연히 "다 하겠습니다"로 시작하지 않는다. 시작 전 숫자로 끝을 정의한다.
- 예: "6개 방 전수 점검 → 6/6 확인되면 종료"
- 예: "파일 12개 수정 → 12/12 커밋되면 종료"
- 진행 중간에 "지금 3/6" 처럼 진행률을 언급해 사용자가 검증 가능하게 한다.
- 종료 판정은 작업 수행자 자신이 아닌 감사실장(또는 사용자)이 한다 — 자기검증 편향 방지.

## 2. 장기 세션 컨텍스트 관리
- 작업이 여러 단계로 길어지면 중간중간 불필요한 과거 로그를 안고 가지 않는다.
- 큰 조사·탐색은 서브에이전트(Explore, research-scout 등)에 위임해 메인 컨텍스트를 깨끗하게 유지한다.
- 한 대화가 명백히 다른 주제로 넘어가면 `/clear` 또는 새 세션을 사용자에게 제안한다.

## 3. 작업 전 보안 체크
- API 키·비밀번호·전화번호 등 민감정보를 코드/커밋에 하드코딩하지 않는다. `.env` + `.gitignore`.
- 여러 저장소를 넘나드는 작업 전 권한 범위(어느 repo까지 접근 가능한지)를 먼저 확인한다.
- 파괴적 작업(강제 push, 삭제, 덮어쓰기) 전엔 항상 사용자 확인.

## 4. 신규 방 연결 시 체크리스트 (배포 브랜치 ≠ 작업 브랜치 재발 방지)

새 저장소를 deploy.yml REPOS에 추가할 때는 배포 성공만 보고 "완료"라 하지 않는다.
1. deploy.yml REPOS 배열에 추가
2. 배포 1회 실행
3. **기본 브랜치가 아니라 그 방의 실제 작업 브랜치를 직접 열어** CLAUDE.md에 지침이 들어있는지 확인
   (배포는 기본 브랜치로 가고, 새로 만든 방의 작업 브랜치는 기본 브랜치와 다를 수 있다 — 실제로 cloud에서 이 문제가 발생함)
4. 어긋나 있으면 배포된 기본 브랜치를 그 작업 브랜치로 병합·push
5. 감사실장 호출로 검증

## 5. 모델 선택 원칙
- 막노동·반복조사 → haiku
- **단순 데이터 조회(파일 읽기, API/상태 확인, git log 확인, 배포 결과 확인 등 판단 없이 결과만 보는 작업) → haiku**
- 계획·코딩·검증 → sonnet
- 정말 복잡한 설계 판단이 필요할 때만 → opus
- 특정 모델의 가격·벤치마크를 단정적으로 인용하지 않는다 — 수치는 계속 바뀌므로 필요 시 사용자에게 직접 확인 요청.
